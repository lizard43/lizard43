Inputs and DIP Switches

The Wizard of Wor input map is not a neat “player 1 port / player 2 port” design. Coin switches, start buttons, joystick leaves, service, tilt, screen flip, Votrax status, and cabinet DIP settings are distributed across the Astrocade input handles and read through the arcade board’s wiring and MAME’s input multiplexer model.

⟵ Video System ⌂ Sound ⟶

Why this chapter matters

The arcade Astrocade input system is easy to misread if you bring normal console assumptions to it. The console idea of “controller 1” and “controller 2” is not enough. On Wizard of Wor-class arcade hardware, the input bytes are a board-level collection of cabinet switches, joystick leaves, service controls, DIP settings, and status bits.

For Wizard of Wor, MAME names the four main readable input handles P1HANDLE, P2HANDLE, P3HANDLE, and P4HANDLE. Those names are useful, but slightly deceptive. P1HANDLE is not “player 1 joystick.” It is the coin/start/service/tilt/flip byte. Player 2 controls are on P2HANDLE. Player 1 controls are on P3HANDLE. DIP switches live on P4HANDLE.

The big idea Treat the input ports as cabinet signal groups, not player objects. Poll raw ports once, decode them into your own clean runtime structure, and only then let game code ask questions like “P1 pressed fire” or “coin edge detected.”

MAME's Wizard of Wor input view

MAME’s WoW input block gives the best compact map for new software work. It shows the ports as four read handles: 

$10  P1HANDLE  coin/start/service/tilt/flip
$11  P2HANDLE  player 2 joystick and buttons
$12  P3HANDLE  player 1 joystick, buttons, unknown bit, Votrax status
$13  P4HANDLE  DIP switch bank S1: coinage, language, lives, bonus, free play, demo sounds
MAME handle Arcade port What it really contains Why the name is dangerous
P1HANDLE $10 Coin 1/2/3, service, tilt, start 1, start 2, flip-screen jumper/DIP. It is not the player 1 joystick.
P2HANDLE $11 Player 2 joystick directions and two button-style inputs. Mostly player 2, but still contains mixed polarity bits.
P3HANDLE $12 Player 1 joystick directions, two button-style inputs, unknown bit, Votrax speech status. Player 1 controls are here, not on P1HANDLE.
P4HANDLE $13 DIP bank S1: coinage, language, lives, bonus life, free play, demo sound. It is configuration, not live player input.
Developer translation Build a decode layer. Never spread raw in a,($10), in a,($11), in a,($12), and in a,($13) tests throughout game logic. Centralize the weirdness once.

How the arcade HVGLIB include names this

The arcade HW-HVGLIB.H.h include intentionally avoids the misleading console assumption that SW0 is always simply “player 1 hand control.” It gives the WoW-style ports more descriptive names: 

PORT_COIN_DOOR   equ $10
PORT_INPUT10     equ PORT_COIN_DOOR
PORT_PLAYER2     equ $11
PORT_PLAYER1     equ $12
PORT_INPUT13     equ $13

SW0              equ PORT_COIN_DOOR
SW1              equ PORT_PLAYER2
SW2              equ PORT_PLAYER1
SW3              equ PORT_INPUT13

This is the right direction: use PORT_COIN_DOOR, PORT_PLAYER2, and PORT_PLAYER1 when writing new arcade code. The old SW0 through SW3 aliases are useful for source compatibility, but they hide meaning.

Generic bit names are useful, but not universal

BIT_UP       equ 0
BIT_DOWN     equ 1
BIT_LEFT     equ 2
BIT_RIGHT    equ 3
BIT_TRIGGER  equ 4
BIT_START1   equ 5
BIT_START2   equ 5

These generic bit positions are handy for a normalized joystick byte, but Wizard of Wor’s raw MAME input block is more complicated. In MAME’s WoW mapping, player direction bits are 0-3, but the primary button-style input is at bit 5, while bit 4 is a separate active-high button-style input. Start 1 and Start 2 are both on P1HANDLE at bits 5 and 6. So do not blindly use the generic masks against raw ports unless your decode layer intentionally made the raw data look like that.

Recommended pattern Use HVGLIB port aliases for hardware access, then decode into your own P1_INPUT, P2_INPUT, COIN_INPUT, and DIP_INPUT workspace bytes. After decoding, the generic BIT_UP/BIT_TRIGGER style names become safe and useful.

P1HANDLE / port $10: coin door, service, tilt, start, flip

This byte is the one that most quickly proves the port names are not player names. On Wizard of Wor, MAME maps P1HANDLE as the cabinet/front-panel/status byte:

Bit Mask MAME meaning Active state Developer note
0 $01 Coin 1 Low Main coin chute / coin-up source.
1 $02 Coin 2 Low Second coin chute.
2 $04 Coin 3 Low Third coin / auxiliary coin input.
3 $08 Service Low Service credit / service switch.
4 $10 Tilt Low Cabinet tilt signal.
5 $20 Start 1 Low One-player start.
6 $40 Start 2 Low Two-player start.
7 $80 Flip Screen jumper / DIP $00 = On MAME notes this as undocumented / jumper-like.

Coin and start edge detection

Coin and start are switch events. You usually want edges, not level. A held coin switch should not add credits every frame. A held start button should not keep restarting the game. 

; Read raw WoW $10.
; Most live switches here are active-low, so invert before edge tests.

ReadCoinStart:
    in   a,(PORT_COIN_DOOR)     ; raw $10 / P1HANDLE
    ld   b,a                    ; keep raw if you need flip/jumper logic
    cpl                         ; now active-low switches become active-high
    ld   c,a                    ; C = normalized active-high cabinet events

    ; Coin 1 edge example:
    ld   a,c
    and  $01                    ; coin 1 active now?
    ; compare against previous normalized coin bit before incrementing credits

    ; Start 1:
    ld   a,c
    and  $20                    ; start 1 active now?

    ; Start 2:
    ld   a,c
    and  $40                    ; start 2 active now?
Important mismatch The generic HVGLIB BIT_START2 value is useful for some normalized/project input layouts, but Wizard of Wor’s raw P1HANDLE start-2 bit is $40, not $20. Decode first.

P2HANDLE / port $11: player 2 controls

P2HANDLE is much closer to what its name sounds like. MAME maps it as player 2 joystick directions and two button-style inputs:

Bit Mask MAME meaning Active state Developer note
0 $01 P2 joystick up Low Direction leaf.
1 $02 P2 joystick down Low Direction leaf.
2 $04 P2 joystick left Low Direction leaf.
3 $08 P2 joystick right Low Direction leaf.
4 $10 P2 button 2 High Different polarity; likely tied to WoW’s special joystick/move behavior in MAME.
5 $20 P2 button 1 Low Primary action-style button in MAME’s map.
6-7 $C0 Unused Low in MAME definition Do not build game logic on these.

Normalize P2 directions

; Decode P2 direction bits into active-high low nibble.
; Result bits: 0=up, 1=down, 2=left, 3=right.

ReadP2Directions:
    in   a,(PORT_PLAYER2)       ; raw $11 / P2HANDLE
    cpl                         ; active-low directions become active-high
    and  $0F
    ld   (P2_DIRS),a

Button handling needs extra care because bit 4 is active-high while bit 5 is active-low. Do not just cpl the whole byte and assume both buttons are now correct.

P3HANDLE / port $12: player 1 controls and Votrax status

On Wizard of Wor, player 1 joystick input is on P3HANDLE, which maps to the arcade include’s PORT_PLAYER1 alias at $12.

Bit Mask MAME meaning Active state Developer note
0 $01 P1 joystick up Low Direction leaf.
1 $02 P1 joystick down Low Direction leaf.
2 $04 P1 joystick left Low Direction leaf.
3 $08 P1 joystick right Low Direction leaf.
4 $10 P1 button 2 High Mixed polarity; do not invert blindly with fire.
5 $20 P1 button 1 Low Primary action-style button in MAME’s map.
6 $40 Unknown Low in MAME definition Leave alone unless a board trace proves its purpose.
7 $80 Votrax speech status High custom Status line from speech hardware/emulation, not a player switch.

Normalize P1 controls safely

; Decode P1 directions and two button-style inputs separately.
; Avoid one big CPL-and-mask for the whole byte because bit 4 and bit 5
; have opposite active states in MAME's WoW input block.

ReadP1:
    in   a,(PORT_PLAYER1)       ; raw $12 / P3HANDLE
    ld   b,a                    ; B = raw

    ; Directions are active-low.
    cpl
    and  $0F
    ld   (P1_DIRS),a

    ; Button 1 is active-low raw bit 5.
    ld   a,b
    cpl
    and  $20
    ld   (P1_BUTTON1),a

    ; Button 2 is active-high raw bit 4.
    ld   a,b
    and  $10
    ld   (P1_BUTTON2),a

    ; Votrax status is active-high raw bit 7.
    ld   a,b
    and  $80
    ld   (VOTRAX_STATUS),a
Why this mattered in our tests “Fire stuck on” or “fire never detected” bugs are easy to create if bit 4 and bit 5 are treated as the same polarity, or if the generic console trigger mask is applied directly to WoW’s raw player port.

P4HANDLE / port $13: DIP switch bank S1

P4HANDLE is the main DIP/configuration byte for Wizard of Wor in MAME. This is where coinage, language, lives, bonus life, free play, and demo-sound behavior are read.

Bit(s) Mask DIP location Function MAME values
0 $01 S1:1 Coin A $00 = 2 coins / 1 credit; $01 = 1 coin / 1 credit.
1-2 $06 S1:2,3 Coin B $04 = 2C/1C; $06 = 1C/1C; $02 = 1C/3C; $00 = 1C/5C.
3 $08 S1:4 Language $08 = English; $00 = foreign ROM required.
4 $10 S1:5 Lives $10 = 2 for 1 credit / 5 for 2 credits; $00 = 3 for 1 credit / 7 for 2 credits.
5 $20 S1:6 Bonus life $20 = after 3rd level; $00 = after 4th level.
6 $40 S1:7 Free play $40 = off; $00 = on.
7 $80 S1:8 Demo sounds $00 = on only when controls are touched; $80 = always on.
Coinage footnote MAME includes a note that if S1:1, S1:2, and S1:3 are all ON or all OFF, only coin meter number 1 will count. That is the sort of operator/cabinet behavior a new game may ignore at first, but a faithful field guide should preserve.

Read DIP settings once, then cache

; DIP switches generally do not need to be polled every frame.
; Read at boot, service menu entry, or attract-mode refresh.

ReadDips:
    in   a,(PORT_INPUT13)       ; raw $13 / P4HANDLE
    ld   (DIP_RAW),a

    ; Example: free play active?
    ld   b,a
    ld   a,b
    and  $40
    jr   z,FreePlayOn           ; MAME WoW: $00 means On
FreePlayOff:
    ; normal credit logic
    ret
FreePlayOn:
    ; bypass credit requirement
    ret

Active states: why raw bits lie

Many cabinet switches are active-low: the idle state reads as 1, and a pressed/closed switch reads as 0. That is why code often does cpl after reading joystick or coin ports. But Wizard of Wor also has active-high bits mixed into the player-control bytes.

Signal group Common active state Exception / trap
Coin, service, tilt, start Active-low Flip-screen/DIP style bit should be treated as configuration, not a momentary switch.
Joystick direction bits 0-3 Active-low Safe to invert and mask $0F for directions.
Button 1 on WoW player ports Active-low bit 5 Invert before testing $20.
Button 2 on WoW player ports Active-high bit 4 Do not invert before testing $10.
Votrax status Active-high bit 7 Status from speech hardware, not a button.
DIP switches Value-coded Do not globally invert; compare masks to documented values.
Rule of thumb Invert only the fields that are known active-low. Do not invert whole bytes and then use every bit as if it became a clean active-high logical input.

A safe polling pattern

The most robust pattern is to read all hardware input bytes once per frame, store the raw values, decode normalized values, then compute edges. This prevents game code from seeing half-old/half-new input states and makes it much easier to debug. 

; Suggested workspace bytes
RAW_PORT10        equ RAM_SCRATCH_START + 0
RAW_PORT11        equ RAM_SCRATCH_START + 1
RAW_PORT12        equ RAM_SCRATCH_START + 2
RAW_PORT13        equ RAM_SCRATCH_START + 3

P1_DIRS           equ RAM_SCRATCH_START + 4
P1_BUTTON1        equ RAM_SCRATCH_START + 5
P1_BUTTON2        equ RAM_SCRATCH_START + 6
P2_DIRS           equ RAM_SCRATCH_START + 7
P2_BUTTON1        equ RAM_SCRATCH_START + 8
P2_BUTTON2        equ RAM_SCRATCH_START + 9

COIN_NOW          equ RAM_SCRATCH_START + 10
COIN_PREV         equ RAM_SCRATCH_START + 11
COIN_EDGE         equ RAM_SCRATCH_START + 12
PollArcadeInputs:
    ; --- Snapshot raw hardware ---
    in   a,(PORT_COIN_DOOR)
    ld   (RAW_PORT10),a

    in   a,(PORT_PLAYER2)
    ld   (RAW_PORT11),a

    in   a,(PORT_PLAYER1)
    ld   (RAW_PORT12),a

    in   a,(PORT_INPUT13)
    ld   (RAW_PORT13),a

    ; --- Decode coin/start byte ---
    ld   a,(RAW_PORT10)
    cpl
    and  $7F                 ; coin/service/tilt/start bits as active-high
    ld   (COIN_NOW),a

    ; edge = now & ~prev
    ld   b,a                 ; B = now
    ld   a,(COIN_PREV)
    cpl
    and  b
    ld   (COIN_EDGE),a
    ld   a,b
    ld   (COIN_PREV),a

    ; --- Decode P1 directions ---
    ld   a,(RAW_PORT12)
    cpl
    and  $0F
    ld   (P1_DIRS),a

    ; --- Decode P1 buttons separately ---
    ld   a,(RAW_PORT12)
    cpl
    and  $20
    ld   (P1_BUTTON1),a

    ld   a,(RAW_PORT12)
    and  $10
    ld   (P1_BUTTON2),a

    ; --- Decode P2 directions/buttons similarly ---
    ld   a,(RAW_PORT11)
    cpl
    and  $0F
    ld   (P2_DIRS),a

    ld   a,(RAW_PORT11)
    cpl
    and  $20
    ld   (P2_BUTTON1),a

    ld   a,(RAW_PORT11)
    and  $10
    ld   (P2_BUTTON2),a

    ret

After this, higher-level code can use COIN_EDGE, P1_DIRS, and P1_BUTTON1 without caring which MAME handle or cabinet wire they came from.

Service-mode reality checks and WoW joystick weirdness

Wizard of Wor has a famously unusual joystick arrangement. The real control is not merely a normal 4-way or 8-way stick. The control has directional sensing and an additional “move/full press” behavior. MAME’s current mapping exposes this awkwardly through the second button-style input, and there is a MAME Testers issue noting that WoW joystick behavior may not be represented perfectly.

What service mode is good for

Service mode can reveal whether the original game thinks a direction, fire, move, coin, or DIP state is active. It is a strong sanity check when MAME behavior and cabinet behavior do not feel the same.

What not to assume

Do not assume a MAME keyboard binding equals the exact physical switch stack under the original joystick. For a new game, you may choose a simpler normalized model, but the field guide should preserve the hardware distinction.

For I, Wizard We can write a clean new-game input layer using directions plus fire, while still documenting the original WoW multi-leaf behavior as a compatibility/hardware note. The important part is not to let MAME’s convenience mapping leak into hardware documentation as if it were the original physical mechanism.

Practical rules for writing robust input code

Do this

Read $10-$13 once per frame and store raw snapshots.
Decode raw ports into clean logical workspace bytes.
Use edge detection for coins and start buttons.
Invert only active-low fields, not whole ports blindly.
Read DIP settings at boot or service-menu refresh.
Keep MAME mapping, service manual wiring, and real cabinet tests separately tagged.

Do not do this

Do not assume P1HANDLE means player 1 joystick.
Do not use console SW0/SW1 assumptions on arcade ports.
Do not globally cpl a player port and then trust every bit.
Do not increment credits from a held coin level every frame.
Do not use raw DIP values without comparing to documented mask/value pairs.
Do not assume Gorf, Robby Roto, Space Zap, and WoW have identical input maps.
