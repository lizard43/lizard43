Why these maps get their own page

The Bally/Midway Astrocade arcade games share the same hardware vocabulary, but they do not share one single address map.

All of these card-rack games live in a Z80 address space. The familiar pieces keep coming back: a low ROM area, a Magic write window, framebuffer RAM, protected bookkeeping RAM, ordinary work RAM, and title-specific ROM or banking hardware. The exact placement changes by game, which is why new code should name addresses by target game instead of assuming that Wizard of Wor, Gorf, Space Zap, and Robby Roto are interchangeable.

Working rule Treat the maps below as the developer-facing model for new Z80 work. When a claim is later verified against schematics, MAME, ROM disassembly, or real board hardware, keep that source note with the map entry.

Machine-model differences across the family

The games share the Astrocade DNA, but the memory maps are not identical. Do not assume Wizard of Wor equals Gorf equals Robby Roto equals Space Zap.

Game / class Memory model notes Why it matters
Wizard of Wor Low ROM/Magic window, video RAM at $4000-$7FFF, extra ROM at $8000-$CFFF, protected RAM at $D000-$D03F, work RAM to $DFFF. Best target for our current replacement-code experiments.
Gorf Same broad Astrocade arcade class, but Gorf has additional speech/sound complexity and its own ROM/board configuration. Good source for routines and patterns, but not a blind drop-in map for WoW.
Space Zap MAME maps low ROM/Magic, video RAM, protected RAM at $D000-$D03F, and a smaller ordinary RAM area ending at $D7FF. Simpler early reference; less RAM headroom than WoW in MAME's model.
Robby Roto MAME maps low ROM/Magic, video RAM, ROM through $DFFF, NVRAM/protected area around $E000, and ordinary RAM at $E800-$FFFF. Valuable commercial-code specimen, but its RAM/protected/NVRAM layout differs heavily from WoW.
Professor Pac-Man / Demons & Dragons Later 16-color/paged variants add banked behavior and screen-RAM control beyond the simpler WoW/Space Zap model. Useful later, but do not contaminate the base WoW machine model with those assumptions.

The shared weird part: low ROM and Magic writes

The most important Astrocade memory-map idea is that $0000-$3FFF has two personalities. On reads, the Z80 sees ROM bytes. On writes, the Astrocade Magic/function-generator path sees a graphics operation.

Read path

ld   a,($0123)
call $0200
jp   $0100

These are ordinary ROM fetches, table reads, calls, and jumps.

Write path

ld   a,$FF
ld   ($0123),a

This does not alter ROM. It feeds the configured Magic write path and affects the video side of the machine.

Developer translation A bad store through a stray pointer may draw garbage instead of crashing immediately. Name low-memory writes as drawing operations in code comments and symbols.

Wizard of Wor memory map

Wizard of Wor is the cleanest working target for the current replacement-code experiments. It has the normal low ROM/Magic region, framebuffer RAM at $4000-$7FFF, fixed upper game ROM, a tiny protected-RAM block, and ordinary work RAM for stack and runtime state.

Stack: place SP at $E000 so pushes grow down into high work RAM Protected RAM: only 64 bytes, so reserve it for credits, latches, and tiny bookkeeping. Magic: reads from $0000-$3FFF are ROM reads; writes are graphics operations.

Gorf memory map

Gorf shares the Astrocade arcade foundation, but the upper map is more complex. The $8000-$BFFF window is a dynamic banked region populated by the Gorf 91364 ROM/RAM board, while high bookkeeping and workspace move upward into the $E000 range.

Z80 address space · Gorf Banked mission code/data window plus high NVRAM and work RAM.

Bank window: the $8000-$BFFF slot is the big difference from Wizard of Wor. NVRAM: Gorf uses high non-volatile/protected storage instead of the WoW-style $D000 block. Port dependency: bank and protected-write behavior must stay tied to the correct I/O model.

Space Zap memory map

Space Zap is the simpler member of this group. It does not need the same upper ROM expansion or dynamic banking used by Wizard of Wor and Gorf, so large parts of the upper address space remain open or unmapped.

Practical coding rules

Safe assumptions

Put a reset jump at $0000.
Initialize SP before calls or interrupts.
Keep stack and large structs in ordinary work RAM.
Use protected RAM only for tiny state.
Name Magic writes separately from direct VRAM writes.

Unsafe shortcuts

Do not write to $0000-$3FFF expecting RAM.
Do not copy the WoW RAM layout into Gorf or Robby Roto.
Do not put object tables in protected RAM.
Do not assume console Astrocade BIOS calls exist on arcade boards.
Do not treat an open-bus region as free RAM.

Suggested symbol style

Keep symbol names explicit enough that a source file says which map it targets. The values below are an example style, not a universal include file.

; Wizard of Wor-oriented names
WOW_ROM_LOW_START       equ $0000
WOW_ROM_LOW_END         equ $3FFF
WOW_MAGIC_WRITE_START   equ $0000
WOW_MAGIC_WRITE_END     equ $3FFF
WOW_SCREEN_RAM          equ $4000
