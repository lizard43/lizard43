Why the pattern board matters

The pattern board is the part of the Astrocade arcade card-rack system that feels the least like a normal console Astrocade program. The home console gives you the Magic system and a framebuffer; the arcade cabinet adds a dedicated card that can move blocks of color-encoded data at high speed.

For Wizard of Wor and Gorf, this board is not a minor helper. It is the hardware path commercial code used for fast pattern transfer: copying patterns from ROM to screen RAM, moving blocks around the screen, and doing it without a slow byte-at-a-time Z80 loop.

Central idea Think of the pattern board as a DMA-like block mover controlled by Z80 OUT instructions. You load parameters into the board through ports $78-$7E. The final height/start write causes the board to request the bus, perform repeated memory read/write cycles, then return control to the Z80.

Working port summary

The short-form map is simple, but the real board behavior is richer than the names imply.

PortHVGLIB aliasWorking roleNotes
$78PBLINADRLPattern board linear/source address low byte.Manual says the low-order eight bits of the 16-bit linear address are loaded here.
$79PBLINADRHPattern board linear/source address high byte.Manual says the high-order eight bits of the 16-bit linear address are loaded here.
$7APBCTLMode/control byte.Manual says this loads U21, the status/control latch. It selects operating modes.
$7BPBROT / PBAREADRLMode-dependent: rotate/line offset/destination low.The manual describes this port as involved with low destination address and line offset. HVGLIB keeps the older/working alias PBROT and also exposes PBAREADRL.
$7CPBMASK / PBAREADRHMode-dependent: mask/destination high.Manual describes this as high destination area address. HVGLIB also names it PBMASK.
$7DPBMR / PBXWIDEWidth / magic register / mode register depending on usage.The manual says width is loaded here. HVGLIB aliases it as PBMR and PBXWIDE.
$7EPBXR / PBYHIGHHeight / X register / transfer start.The manual says height is loaded here and that this output also starts the actual pattern transfer sequence.
$7FPBORPossible OR/output behavior; needs confirmation for WoW/Gorf programming use.MAME's classic pattern-board map focuses on $78-$7E. HVGLIB includes PBOR at $7F. Treat as an investigation target until tied to schematic/code behavior.
$A55BPROTECT via $5B + A5Protected RAM enable path, separate from the pattern board itself.Important because MAME maps protected RAM enable near the same hardware discussion. Do not confuse it with pattern transfer parameters.
Vocabulary caution The same physical ports are described through several vocabularies: MAME map, HVGLIB aliases, and the service manual's hardware/counter language. The safest guide language is: port, alias, observed role, and confidence tag.

What the Wizard of Wor manual says

The uploaded Wizard of Wor manual includes a Pattern Board Test and a Pattern Board Theory of Operation section around the rendered PDF pages 96-100, printed manual pages 49-53. The important programmer-facing pieces are:

Manual sectionUseful developer translation
Pattern Board TestA good pattern card displays LOOKS GOOD; a bad card displays the area of trouble. The test procedure names the card-rack board tester, test PROMs, and pattern card connector flow. This confirms the board was treated as independently testable hardware.
Fatal error tableThe manual identifies data latches, flush data bit, direction data bit, area/linear high bits, tri-state buffers, and control logic clock as major failure groups.
U21 status latchU21 controls direction, expand, constant, flush, flip, and flop style behavior. The manual says this chip controls shifting data in a direction, making larger objects, moving same-size objects, and turning/rotating objects.
Theory of operationThe board's function is to move blocks of data from one memory location to another quickly. In the card-rack system, that means transferring a pattern on the screen, or from ROM, to another location on the screen.
Parameter loadingThe manual explicitly says parameters are passed to the pattern board by output instructions to ports 78 through 7E.
Bus takeoverThe output to port 7E starts the sequence by causing BUSREQ to go low. The Z80 tri-states address, data, and control signals and acknowledges with BUSACK.

The transfer sequence

The manual's description is a very useful mental model for a programmer. It reads like a DMA transaction even if the manual never uses that word.

; Conceptual pattern-board setup, not a final drop-in API.
; Names are HVGLIB-friendly; meanings are based on manual + MAME + experiments.

    ; 1. Select mode/control behavior.
    ld   a,mode_bits
    out  (PBCTL),a        ; $7A, U21 control/status latch

    ; 2. Load 16-bit source / linear address.
    ld   a,source_low
    out  (PBLINADRL),a    ; $78
    ld   a,source_high
    out  (PBLINADRH),a    ; $79

    ; 3. Load destination / area address or related line/rotate/mask state.
    ld   a,dest_low_or_line_offset
    out  (PBROT),a        ; $7B, also PBAREADRL in HVGLIB
    ld   a,dest_high_or_mask
    out  (PBMASK),a       ; $7C, also PBAREADRH in HVGLIB

    ; 4. Load width.
    ld   a,width_count
    out  (PBMR),a         ; $7D, also PBXWIDE

    ; 5. Load height and start the transfer.
    ld   a,height_count
    out  (PBXR),a         ; $7E, also PBYHIGH - starts transfer

What happens after the start write

PhaseBoard behaviorWhy it matters
StartPort $7E write causes BUSREQ to go low.The Z80 is about to lose the bus temporarily.
Bus acknowledgeZ80 tri-states address/data/control lines and asserts BUSACK.The pattern board can now generate bus cycles that look like Z80 cycles.
Read cycleThe first cycle reads from the address in the linear/source address counters; data lands in data hold registers U19 and U10.This is the ROM/screen source fetch.
Write cycleThe next cycle writes to the address in the area/destination counters; data is driven through tri-state buffers U20 and U11.This is the destination screen/RAM write.
Horizontal countRead/write cycles continue until width counters reach zero.One horizontal line of the pattern has been moved.
Next lineWidth is reloaded; height decrements; the line offset is added to the current destination/start for the next line.This is how rectangular shapes stride through screen memory.
DoneWhen height reaches zero, BUSREQ is released and all board-driven address/data/control lines tri-state.Z80 regains normal bus control and execution continues.

Control and status bits

The manual's Pattern Board Test page identifies U21 as a 74LS174 status latch and gives named pin functions. This is one of the most useful bridges between the service schematic and what a programmer sees as the PBCTL byte.

U21 pinManual labelProgrammer meaningConfidence
2DirectionControls direction of shifting/movement during board operation.manual-confirmed
5ExpandControls expansion / making a larger object.manual-confirmed
7ConstantLikely uses a constant/fill value rather than ordinary source data in some modes.manual-confirmed label, behavior to test
10FlushThe manual says flush sets memory to a fixed color.manual-confirmed
12FlipObject transform / orientation control.manual-confirmed label
15FlopObject transform / orientation control.manual-confirmed label
Connection to the Astrocade Magic vocabulary Direction, expand, flip/flop, and flush sound very close to Magic-style operations, but the pattern board is external card-rack hardware. Do not automatically treat the pattern board's control byte as identical to the Astrocade MAGIC register at port $0C. Similar concepts, different hardware path.

The alternate aliases are helpful because they admit the messy truth: $7B and $7C are not just "rotate" and "mask" in every useful explanation. In the Wizard manual theory, those same ports participate in the area/destination address and line offset side of the transfer.

The $7F question: PBOR

The requested map includes $7F as OR? / related behavior if confirmed from schematic or code. That is exactly how this guide should treat it.

EvidenceWhat it saysGuide stance
HVGLIBDefines PORT_PBOR equ $7F and alias PBOR.Good enough to preserve as a named symbol.
MAME classic mapPattern-board writes are usually discussed as $78-$7E for the classic games.Do not claim $7F is part of the core documented setup sequence until verified.
Wizard manual theory pageThe parameter sequence explicitly says ports 78 through 7E.Manual does not make $7F part of the standard transfer-start sequence.
Needed nextTrace original WoW/Gorf writes to $7F, inspect schematic net names around the port decode, and compare MAME behavior.Mark $7F as working hypothesis until proven.
Documentation rule Keep PBOR in the symbol table. Do not teach it as a required pattern-transfer parameter unless we can tie it to original ROM usage, MAME behavior, or the schematic.

Later variants: Professor Pac-Man and Demons & Dragons

MAME's Astrocade driver treats the later 16-color/paged hardware as related but not identical to the simpler Wizard/Gorf/Robby/Space Zap card-rack model. Professor Pac-Man and Demons & Dragons add banked behavior, screen-RAM control, and intercept/paging concepts that should not be backported into this WoW/Gorf chapter.

Hardware familyPattern-board lesson
Space Zap / WoW / Gorf / RobbyClassic CPU + Pattern Board + RAM Board card-rack model. This chapter targets this group.
Professor Pac-Man / Ten Pin DeluxeUses later 16-color CPU/screen RAM/super game memory/pattern mover configuration. Related concepts, different assumptions.
Demons & DragonsRelated later Astrocade hardware with banking/intercept behavior in MAME. Useful later, but not the base WoW pattern-board API.

Code recipes and test scaffolds

These are deliberately conservative. They show the write order and how to instrument the board without pretending we already have every mode byte pinned down.

Minimal transfer skeleton

; Inputs:
;   HL = source / linear address
;   DE = destination / area address
;   B  = width count
;   C  = height count
;   A  = mode bits before call, or load mode inside routine
;
; This is a documentation skeleton. Real code should preserve registers as needed
; and use known-good mode/stride values from ROM traces or tests.

PatternMoveSkeleton:
    out  (PBCTL),a        ; $7A mode/control

    ld   a,l
    out  (PBLINADRL),a    ; $78 source low
    ld   a,h
    out  (PBLINADRH),a    ; $79 source high

    ld   a,e
    out  (PBAREADRL),a    ; $7B destination low / area low / line-offset-related
    ld   a,d
    out  (PBAREADRH),a    ; $7C destination high / area high / mask-related

    ld   a,b
    out  (PBXWIDE),a      ; $7D width
    ld   a,c
    out  (PBYHIGH),a      ; $7E height and start
    ret

Debugging write trap

; In MAME debugger, catch any pattern-board activity:
;   wp 0:io,78,8,w
;
; Or watch one start port:
;   wp 0:io,7e,1,w
;
; Then inspect PC, A, and nearby code to learn original ROM calling convention.

Recommended trace labels

PB_LAST_MODE       equ scratch+0
PB_LAST_SRC_LO     equ scratch+1
PB_LAST_SRC_HI     equ scratch+2
PB_LAST_DST_LO     equ scratch+3
PB_LAST_DST_HI     equ scratch+4

For our own ROMs, it is worth shadowing every parameter in scratch RAM before writing the ports. Then a crash or graphical mess can be compared against the expected transfer setup.

Rules of thumb

Do this

Use PBLINADRL/PBLINADRH for the source address.
Comment $7B and $7C with the specific role used by that routine.
Treat the $7E write as the transfer-start point.
Shadow parameters in RAM before firing the board.
Build a MAME watchpoint recipe for all $78-$7F writes.
Tag every mode-byte claim by source: manual, MAME, ROM trace, or real hardware.

Do not do this

Do not assume the pattern board is the same as the Astrocade MAGIC register.
Do not assume PBROT always literally means rotate.
Do not write $7E until every other parameter is loaded.
Do not claim $7F semantics until verified.
Do not copy Professor Pac-Man 16-color assumptions into WoW/Gorf code.
Do not run pattern-board experiments on real hardware without a known-safe setup and reset path.
