91354 at a glance

The 91354 CPU Board is the timing and execution center of Bally/Midway's early Astrocade-derived commercial card-rack system.

It is not a complete game board by itself. It plugs into the A082-93006-C/D000 back-panel motherboard and shares the rack bus with the 90708 game board, ROM/RAM board, RAM cards, and the 91355 Pattern Board.

The short version for bench work: 90708 carries the cabinet/game I/O custom chips, while 91354 carries the Z80, Custom Address chip, Custom Data chip, master crystal, memory-cycle timing, video timing, and rack-bus arbitration logic.
Board assembly A082-91354-F000 CPU Board / CPU Card.
Main processor Z80 at approximately 1.7897725 MHz.
Custom ICs Custom Address at U3; Custom Data at U15.
Clock source 14.31818 MHz crystal, divided into video/system/CPU clocks.
Rack role Normal bus master, but can yield the bus through BUSREQ/BUSACK.
Location cheat The two Custom I/O chips are on the 90708 Game Board. The Custom Address and Custom Data chips are on this 91354 CPU Board.

Card-rack role

The 91354 is the orchestrator of the reusable six-card Z80 rack. The game-specific board decides what cabinet and rules the stack belongs to; the 91354 decides how the Z80, RAM timing, custom video logic, and rack bus stay in phase.

CPU execution

The Z80 executes game code from the ROM/RAM card and sees cabinet inputs through the 90708 custom I/O path.

Memory arbitration

The Address chip arbitrates normal CPU memory cycles against the scan address generator that continuously feeds the CRT raster.

Video timing

The Data chip generates horizontal and vertical drive timing, feeds the serialized pixel path, and participates in VIDEO, R-Y, and B-Y output generation.

Pattern-board cooperation

The 91355 Pattern Board can request the bus. The Z80 yields with BUSACK, letting pattern hardware move memory blocks at raw hardware speed. In the documented pattern-board sequence, a write to port $7E starts the transfer and asserts the bus request; width is loaded through $7D and height through $7F, so $7E should not be labeled as the height register.

Master clock tree

The board derives its timing from a single 14.31818 MHz crystal. That frequency is four times the NTSC colorburst frequency, which makes sense because this board is not just a CPU card; it is also a video timing card.

Crystal master

Y1 is the 14.318 MHz crystal shown on the component layout. Everything important is divided down from this timing reference.

Chroma reference

The back panel exposes CHROMA at approximately 3.579545 MHz, the NTSC subcarrier rate.

System clocks

7M and 7M̄ run at about 7.15909 MHz and feed the custom Address/Data timing domain.

Z80 phase clocks

The Z80 clock Φ and opposite phase Φ̄ are about 1.7897725 MHz, one quarter of the 7M domain.

Why the odd CPU speed? The CPU speed is a consequence of the video-derived master clock. The board trades a nice round CPU frequency for rock-solid alignment between CPU cycles, memory scan cycles, color/video timing, and custom-chip write windows.
Two-phase timing matters The rack exposes both Φ and opposite-phase Φ̄. That is a hardware-timing aid, not an extra CPU feature. The custom Address/Data logic uses the divided video clock domains to keep Z80 bus activity, RAM access, and raster fetches from colliding. For software, the visible symptom is that memory access can be stretched by hardware wait states when the display side has priority.

Custom Address and Custom Data chips

The most important repair and programming lesson is that the Z80 is not directly in charge of everything. The custom chips sit between Z80 intent, memory timing, video scan timing, and the final memory-data bus.

U3 Custom Address chip

The Address chip contains the microcycle decode and scan-address machinery. It turns Z80/microcycle information into RAM-side address timing and coordinates RAM access with the video scan generator.

WAIT-state insertion

If the scan address generator and the Z80 need RAM at the same time, the scan generator wins. The Address chip can assert WAIT so the raster stays stable.

U15 Custom Data chip

The Data chip handles TV sync generation, parallel-to-serial video shifting, microcycle control bits, and the memory data path.

Magic Function Generator

During memory write cycles, the Data chip can modify CPU-side MUXD0-MUXD7 data before placing it on MD0-MD7. This is the hardware basis for Astrocade Magic writes: hardware-controlled shift, merge, mask, expand, and logical effects can happen as bytes cross into RAM.

Factory-described data flow

  • Address chip: Generates Z80-derived address bits, scan addresses, RAM select timing, and memory-cycle control.
  • Scan priority: The scan address generator can hold the Z80 for a short wait slot so video fetches remain synchronized to the CRT.
  • Data chip: Loads parallel memory data, shifts serial video, produces timing signals, and drives the Magic write path.
  • Magic writes: Hardware can shift/merge/modify Z80 write data before it reaches RAM, which is why software can draw faster than a plain byte-copy loop.

Back-panel interconnect

The 91354 edge connector exposes CPU control, memory buses, custom-chip timing, video outputs, clocks, reset, and power rails to the shared back panel. This is why a CPU-board problem can look like a RAM-card problem, and a RAM-card problem can look like a video problem.

Gate-level schematic notes

The 91354 looks simple from a distance: one Z80, two custom chips, and support TTL. The schematic shows why it is actually the collision point for CPU, RAM, video, and pattern-transfer timing.

Bus transceiver

U21 is a 74LS245 octal bus transceiver around the Z80/custom-chip data path.

Multiplexers

U12 and U16 are 74LS257 multiplexers; U13 is a 74LS157. These support the address/data selection logic.

DRAM timing

RAS0-RAS3, CAS, memory data, and multiplexed address signals are generated/coordinated by the CPU-board custom logic and TTL glue.

Analog video

The schematic includes VIDEO, R-Y, B-Y, and the 2.5V reference path, so video-output failure is not isolated to a monitor or separate video card.

Pattern Board bus seizure The Pattern Board's transfer engine requests the bus with BUSREQ. The Z80 answers with BUSACK and tri-states its address/data/control outputs so pattern hardware can move RAM data without normal CPU copying. In factory terminology, this transfer is started by the Pattern Board control sequence; $7E is the start/bus-request trigger, while width and height are loaded through the surrounding pattern-board ports such as $7D and $7F.

Software implications

When writing or debugging Z80 code for Wizard of Wor-class hardware, the 91354 explains several behaviors that feel strange if you are expecting a plain Z80 RAM map.

  • VRAM writes can wait: The scan generator has priority over the CPU for display fetches. A display-memory write loop can receive short wait-state delays, so cycle counts are not perfectly fixed when code hammers display RAM.
  • Magic is CPU-board hardware: Magic write effects come from the Custom Data chip path on the 91354, not from the Pattern Board. If Magic is enabled, the byte stored in RAM may not be the same byte the Z80 placed on the bus.
  • Pattern transfers are separate: The 91355 Pattern Board accelerates block movement by taking bus ownership, not by executing more Z80 instructions. Large transfers steal real CPU time.
  • Register state is part of the graphics state: Code must know the current Magic/expand/rotate/shift mode before doing display writes. A forgotten mode bit can turn an ordinary byte copy into a shifted or merged write.
  • Active-low notation matters: Factory signal names often use overbars. MREQ, IORQ, WR, RD, and related lines need to be read as asserted-low signals.

Programmer timing model

For a systems programmer, emulator author, or anyone writing new Z80 code for Wizard of Wor-class hardware, the 91354 is very different from a clean Z80 computer. The CPU is only one participant in a synchronous video-and-memory machine.

Corrected mental model Do not think of this as “Z80 + RAM + video buffer.” Think of it as Z80 + custom Address chip + custom Data chip + scan-address generator + optional Pattern Board bus master. Software and hardware take turns touching the same memory system.

1. Cycle stealing by the Pattern Board

The 91355 Pattern Board is a hardware block-transfer engine. Its documented register sequence loads source, destination, width, height, direction/mode, and related parameters through ports $78-$7F. Width is loaded through $7D, height through $7F, and the transfer is kicked by the Pattern Board start path; on the documented theory-of-operation page, the $7E write is the operation-start point that pulls BUSREQ low.

Once the Z80 acknowledges with BUSACK, it tri-states address, data, and control signals. During that interval, the Pattern Board is not “running code faster”; it owns the bus while the Z80 is temporarily stopped. That is great for moving graphics, but bad for anything that expects uninterrupted instruction timing.

Good use

Bulk pattern movement, fills, object shifts, and screen-area updates that would be wasteful as pure Z80 byte loops.

Timing caution

A large transfer can delay CPU-side work. Avoid scheduling huge Pattern Board operations across tight sound streaming, watchdog-sensitive code, or code that assumes exact instruction timing.

2. Magic writes mutate the write path, not the read path

The Custom Data chip can transform bytes during memory write cycles. This is the reason the Astrocade family can draw shifted, expanded, masked, or logically merged graphics without spending piles of Z80 cycles on SRL, RLC, and hand-built mask loops.

The important correction: reading RAM does not magically mutate the byte on the way back. The stored byte is the stored byte. The trap is that the stored byte may differ from the original source byte because the write went through the active Magic/expand/shift mode. Software has to track the current hardware graphics mode as part of its drawing state.

Hardware path Software-visible effect Programming consequence
Magic control Changes how CPU write data is merged or transformed before reaching RAM. Reset or explicitly set Magic state before general-purpose memory copies.
Expand / shift state Lets byte-oriented sprite data land at useful pixel alignments without full software bit shifting. Fast sprite drawing depends on setting the hardware state once, then streaming bytes.
RAM readback Returns the byte that actually got stored. Useful for verification, but do not expect it to equal the unmodified source byte after a Magic write.

3. Wait states make cycle counts conditional

The Address chip gives video scan access priority over ordinary CPU memory cycles. The factory text says the scan address generator never requires memory for more than one consecutive memory cycle, so the Z80 should not be held off in long scan bursts. Still, display-memory-heavy code is not a perfect fixed-cycle environment.

  • Plain RAM math: Code that stays out of contested display fetch areas behaves much more like normal Z80 code.
  • VRAM loops: Repeated screen writes may pick up short hardware wait states depending on where the raster and scan generator are.
  • Emulation note: A forgiving emulator can run many games with coarse timing, but cycle-accurate rendering and tricky raster behavior require modeling the scan/WAIT relationship.

4. Protected RAM enable is an alternating enable/write pattern

The protected-RAM mechanism is a platform-level guard latch, not normal writable RAM. In code and in MAME terminology it is often described as a protected RAM enable write at $A55B. In Z80 source, the usual way to create that I/O cycle is to load A=$A5 and execute OUT ($5B),A; the immediate port is $5B, while A also appears on the high half of the Z80 I/O address bus.

; Protected RAM write pattern.
; A=$A5 enables the latch; the following protected-RAM write consumes it.

        ld   a,$a5
        out  ($5b),a        ; I/O address appears as $A55B on the Z80 bus
        ld   a,value1
        ld   ($d000),a      ; first protected byte write

        ld   a,$a5
        out  ($5b),a        ; enable again for the next protected write
        ld   a,value2
        ld   ($d001),a      ; second protected byte write
Practical warning Do not treat protected RAM as a block-copy target. If the hardware or emulator is enforcing the latch correctly, only the write immediately following the enable sequence should land. Later writes without another enable are ignored.

5. Interrupts are tied to the video rhythm, but not every sync line is an interrupt

The Custom Data chip generates horizontal and vertical drive timing for the rack, and the back panel exposes HORZDR and VERTDR as timing signals. The Z80 also exposes INT and NMI on separate back-panel pins. It is safer to describe the software model as frame-synchronized, not as “HORZDR and VERTDR are simply hardwired to INT and NMI” in every board context.

For game code, the operational point remains the same: the main loop is paced by the video frame. If too much work is packed into one 16.6 ms frame budget, input response, audio updates, object motion, and display updates can slip into the next frame.

Diagnostic and maintenance notes

The factory test notes are useful because they describe symptoms in terms a tech would actually see: black screen, flashes, random bars, rainbow pattern, garbage output, or a missing clock.

25-second trap

In test paths, the program can spend roughly 25 seconds cycling through screen RAM, static RAM, and ROM checks before returning to a visible sequence. A black screen immediately after reset is not always instant death.

Random bar clues

Screen RAM trouble is described as random vertical bars. Static RAM trouble is described as shorter bursts of horizontal bars.

Rainbow pattern path

The card-rack tester uses a rainbow-pattern display to prove enough of the video path is working. Garbage instead of clean bars points toward video-output logic.

Strap mapping

JU1-JU9 straps are real decode configuration, not cosmetic jumpers. ROM-size and mapping changes can require physical cut/solder changes.
