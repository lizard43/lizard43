91355 at a glance

The 91355 Pattern Transfer Board is the card-rack system's external pattern mover. It copies rectangular blocks of bytes from one memory area to another fast enough for arcade animation, while the Z80 temporarily gives up the bus.

In normal game terms, this is the board that lets Wizard of Wor, Gorf, Space Zap, and related card-rack titles move graphics without forcing the CPU to hand-copy every byte of every sprite, tile block, or screen region. Software sets up a source address, destination address, mode byte, width, and height. The board then becomes the bus master and performs the read/write cycles itself.

Most important correction The manual does not describe port $7C as a byte-stream graphics gateway. In the Pattern Board theory notes, $7C loads the high byte of the area/destination address. The transfer is started by writing the height register at $7E, which asserts BUSREQ and makes the Z80 release the bus.
Programming consequence This is not a clean, isolated Z80-only graphics API. Code that uses the 91355 is coordinating with synchronous hardware. During the transfer window, the CPU is stopped, the Pattern Board drives the address, data, and control busses, and the transfer timing is determined by the board's counters and clock logic.

Card-rack role

The commercial card-rack assembly lists the Pattern Transfer Board as A082-91355-C000. It sits between the CPU/memory system and the graphics workload: the CPU supplies transfer parameters through I/O ports, and the board generates memory cycles that look like Z80 read/write cycles while it owns the bus.

Source side

The manual calls this the linear address. It is usually the source address: pattern data in ROM, RAM, or another region of memory that will be read into the Pattern Board's data latches.

Destination side

The manual calls this the area address. It is usually the destination address: screen RAM or a work area that receives the copied and transformed data.

Shape control

Width and height counters let the board repeat read/write cycles across a rectangular block, then step to the next horizontal line until the height count expires.

Mode control

The U21 status latch selects behaviors such as direction, expand, constant, flush, flip, and flop. These modes are the board's hardware-side pattern manipulation controls.

Pattern Board I/O port map

The Pattern Board is programmed with output instructions to ports $78 through $7E. The order matters because the final write to $7E both loads the height and starts the transfer.

Port Manual function Hardware latches / counters Practical programming meaning
$78 Linear address low byte U34 and U25 Low byte of source address.
$79 Linear address high byte U16 and U7 High byte of source address.
$7A Mode / status control byte U21 Selects transfer behavior: direction, expand, constant, flush, flip, flop.
$7B Area address low byte / line offset value U39 and U30 Low byte of destination address and line stepping value used between rows.
$7C Area address high byte U18 and U9 High byte of destination address. This is not a data-stream portal.
$7D Pattern width U41 and U32 Horizontal byte count for each transfer line.
$7E Pattern height and transfer start U40 and U31; U13 starts BUSREQ sequence Vertical line count. The write also starts the bus-master transfer.
Do not mix this up with Astrocade video custom-chip ports The color registers, color-map block load, video mode, palette split, scanline interrupt, and Magic function live in the CPU/video custom-chip I/O space. They are part of the same machine, but they are not the 91355 Pattern Board register block.

Transfer pipeline and bus ownership

The manual's theory section describes the 91355 as a hardware read/write sequencer. Once started, the board performs a read cycle from the linear/source address, stores the byte in data hold registers, then performs a write cycle to the area/destination address using tri-state data buffers.

1. Parameter load

The Z80 writes the mode byte, source address, destination address, line offset, width, and height. The height write is the trigger.

2. BUSREQ asserted

The $7E write presets one half of U13 and sends BUSREQ low. The Z80 responds by tri-stating address, data, and control signals, then asserting BUSACK low.

3. Pattern Board drives the bus

With BUSACK low, the Pattern Board enables buffers for RD, WR, MREQ, and IORQ, enables address/data multiplexers and data buffers, and starts the U5 master clock generator.

4. Read/write pairs repeat

The first cycle reads from the linear address into U19 and U10. The next cycle writes to the area address through U20 and U11. This continues until the width counter reaches zero.

5. Next line setup

The width count reloads. The height count decrements. Adders U38 and U29 combine the line offset with the current area count so the next horizontal row starts at the proper address.

6. Return bus to Z80

When the height counters reach zero, BUSREQ returns high, Pattern Board outputs are tri-stated, and the Z80 takes BUSACK high to continue normal execution.

U21 status latch

U21 is a 74LS174 status latch. The diagnostic pages explicitly identify several status outputs by pin. They control how the transfer hardware shifts, expands, flushes, flips, flops, or changes direction while moving data.

U21 pin Status function Diagnostic clue
Pin 2 Direction The test page calls out Direction Data Bit 01 at U21 pin 2.
Pin 5 Expand Used for larger-object / expanded-data behavior.
Pin 7 Constant Used with fixed-data / constant transfer behavior.
Pin 10 Flush The test page calls out Flush Data Bit 08 at U21 pin 10. Flush sets memory to a fixed color/value.
Pin 12 Flip Part of the board's pattern orientation logic.
Pin 15 Flop Part of the board's pattern orientation logic.
Bit-value caution The manual pages shown here explicitly tie $01 to Direction and $08 to Flush. They list the other U21 functions by pin, but do not fully document a friendly programmer bitfield for every mode combination. When lifting code from a ROM or emulator symbol table, verify the actual mode byte against the schematic and the disassembly path that uses it.

Color and video interaction

The 91355 moves and transforms bytes. It does not own the master color palette. After a byte lands in screen RAM, the Astrocade video hardware interprets that byte as packed pixel color indices and maps those indices through the active palette registers.

2-bpp screen bytes

In high-resolution mode, one byte represents four pixels. The two-bit groups select color indices 0 through 3. A byte such as %01101100 maps to four pixels: 01, 10, 11, 00.

Palette split

The machine has left/right palette behavior controlled outside the 91355. The video system can use one set of color registers on one side of the horizontal boundary and another set on the other side.

Color-map block load

At the system level, software commonly writes the eight palette bytes through the color-map block-load path at $0B. That is a video/color operation, not a Pattern Board transfer register.

Expansion modes

The U21 Expand function is real, but avoid describing $7B or $7C as a simple foreground/background color-mask port unless the exact game code and board mode have been traced.

Programming notes and corrected Z80 setup

A useful software model is: load the 91355 registers, start the transfer with OUT ($7E),A, then assume the CPU will not execute again until the board releases BUSREQ. The source data is not streamed through an output port. The board reads it from the source address itself.

; =============================================================================
; Bally/Midway 91355 Pattern Transfer Board - corrected register map
; =============================================================================
PORT_PB_LIN_LO   EQU $78    ; Linear/source address low byte
PORT_PB_LIN_HI   EQU $79    ; Linear/source address high byte
PORT_PB_MODE     EQU $7A    ; U21 status/mode latch
PORT_PB_AREA_LO  EQU $7B    ; Area/destination low byte and line offset latch
PORT_PB_AREA_HI  EQU $7C    ; Area/destination high byte
PORT_PB_WIDTH    EQU $7D    ; Width in bytes
PORT_PB_HEIGHT   EQU $7E    ; Height in rows; write starts transfer

VRAM_START       EQU $4000
SCREEN_STRIDE    EQU 80      ; 320 pixels / 4 packed 2-bpp pixels per byte

; Draw an 8x8 2-bpp object. 8 pixels = 2 bytes per row, 8 rows high.
; Source is read by the Pattern Board from MonsterGraphic2bpp.
; Destination is screen RAM at row 64, byte-column 31.

DrawMonster91355:
    ; 1. Mode byte. 0 is shown here as the conservative/default transfer setup.
    ;    Use real ROM/schematic-verified mode values for Direction/Expand/Flush/etc.
    XOR A
    OUT (PORT_PB_MODE), A

    ; 2. Load linear/source address.
    LD HL, MonsterGraphic2bpp
    LD A, L
    OUT (PORT_PB_LIN_LO), A
    LD A, H
    OUT (PORT_PB_LIN_HI), A

    ; 3. Load area/destination address.
    LD DE, VRAM_START + (64 * SCREEN_STRIDE) + 31
    LD A, E
    OUT (PORT_PB_AREA_LO), A
    LD A, D
    OUT (PORT_PB_AREA_HI), A

    ; 4. Load shape. The height write starts BUSREQ/BUSACK transfer.
    LD A, 2                   ; two bytes per row = 8 pixels at 2-bpp
    OUT (PORT_PB_WIDTH), A
    LD A, 8                   ; eight rows; this OUT starts the transfer
    OUT (PORT_PB_HEIGHT), A
    RET

MonsterGraphic2bpp:
    DEFB %00001111, %11110000
    DEFB %00111111, %11111100
    DEFB %11110011, %11001111
    DEFB %11000000, %00000011
    DEFB %11001111, %11110011
    DEFB %11110011, %11001111
    DEFB %00111111, %11111100
    DEFB %00001111, %11110000
Interrupt timing note If you start transfers from the main loop, the Z80 will be suspended while the board owns the bus. If the transfer is visible or long enough to fight the raster, schedule it during the vertical blank / bottom-of- visible-screen service path used by the rest of the game, not randomly in the middle of active drawing.

Diagnostic and fault testing

The Pattern Board test procedure uses the Card Rack Board Tester and Pattern Test PROMs. A good Pattern Card displays “LOOKS GOOD”. A bad Pattern Card displays the area of trouble on the video screen.

Pattern Board tester setup

  • Tester card: Move the Card Rack Board Tester Card A080-91517-A000 from J-4 to J-6.
  • Test ROMs: Remove the RAM Test PROMs from X-4 and X-5, then install the two Pattern Test PROMs in those locations.
  • Video: Connect the video cable to Card Rack Board Tester connector J-2, pins 11 and 12.
  • Card under test: Insert the Pattern Card into the 50-pin connector on the Card Rack Board Tester, then press the red reset button.
  • Required RAM: The manual notes that a proper Pattern Test requires two RAM Cards PC A082-91356-B000.

Runtime symptom translation A totally dead transfer path can leave the game in a black-screen or boot-loop condition because early system initialization and RAM/video tests depend on valid memory movement. Partial failures are more likely to show as displaced strips, repeated bars, missing sprite sections, bad masks, or objects that corrupt nearby screen memory as they move.

Emulation and systems-programming notes

For an emulator or low-level software model, the Pattern Board should be treated as a bus-mastering transfer engine, not as a CPU-side pixel parser. The Z80 writes parameters, the board steals the bus, and the transfer consumes hardware cycles while the CPU is effectively paused.

Emulate BUSREQ timing

At minimum, code that writes $7E should account for a transfer delay proportional to width, height, and per-byte read/write cycles. Ignoring the pause makes code appear faster and less raster-bound than real hardware.

Model address stepping

The line offset and feedback/add logic matter. A one-row copy is simple; a rectangular blit must reload width, decrement height, and derive the next area address correctly.

Separate video color from transfer

The 91355 moves bytes. Palette registers and horizontal palette split are separate system-level video controls. The same copied byte can display differently depending on palette state.

Preserve failure modes

Bad data latches, bad tri-state buffers, or broken high address bits produce very different visual symptoms. Emulation-level diagnostics are easier if those sub-blocks are named the same way the manual names them.
