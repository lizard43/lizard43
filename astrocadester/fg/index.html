Why this guide exists

Bally Alley is invaluable, and the console Astrocade material is rich, but the arcade side has a hole right where a developer wants the most help. There are service manuals. There is MAME. There are schematics, patents, disassemblies, and reverse-engineering notes. What does not seem to exist is a friendly field manual that says: here is how you write a new arcade ROM for this hardware.

Goal: build the missing middle layer between technician documentation and working software: a practical, tested, source-backed guide for new Astrocade arcade development.

This first page is the front door. It names the terrain, sets the rules, and points to the deeper weeds: memory maps, I/O ports, Magic writes, color maps, pattern board registers, sound, interrupts, inputs, and the many places where console Astrocade knowledge helps but cannot be copied blindly.

The journey

The path starts with the obvious question: why is this so hard? The Astrocade arcade platform was not a consumer development target. Bally/Midway and Dave Nutting Associates built commercial embedded games, shipped ROMs, and documented the machines mainly so operators could keep them running.

The result is a weird archaeological dig. The service manual tells you what the board is. MAME tells you how the emulator sees it. The console manuals tell you what the custom chips were meant to do. Robby Roto and Wizard of Wor disassemblies tell you how commercial games actually used them. Real hardware tells you which assumptions survived contact with the cabinet.

Not just console Astrocade

The home system knowledge matters, especially Magic, palette, and sound concepts. But arcade machines add card racks, different input routing, pattern boards, protected RAM behavior, speech hardware, and hi-res video assumptions.

Not just MAME either

MAME is the best executable specification we have, but this guide should distinguish emulator-confirmed behavior from schematic-confirmed behavior and from real-hardware-confirmed behavior.

Source map

No single source is enough. This guide treats each source as a different lens on the same machine.

SourceBest useDanger
Bally Alley Master index, scans, historical notes, console and arcade cross-links. Not a single coherent arcade programming manual.
MAME arcade driver Memory maps, I/O maps, input definitions, emulated behavior. Can hide analog/electrical quirks and real-board timing surprises.
MAME Astrocade sound device Sound register behavior and arcade SI/SO caveats. Sound output recipes still need practical arcade examples.
Service manuals Schematics, wiring, card racks, DIP switches, edge connectors, board layouts. Written for repair, not new software development.
Robby Roto paper and disassembly Commercial software architecture on related hardware. Game-specific engine choices can look like hardware rules.
Console Astrocade docs Magic, palette, sound, Z80 conventions, original chip concepts. Home-system assumptions can be wrong on arcade card-rack hardware.
Our tests Minimal examples, MAME debugger traces, real WoW hardware behavior. Must be labeled clearly until reproduced and verified.

The arcade Astrocade family

The family is broader than Wizard of Wor and Gorf. The same design lineage appears across multiple Bally/Midway titles, with variations in ROM layout, RAM, input routing, speech, and pattern-board usage.

This guide will begin with the Wizard of Wor / Gorf / Robby Roto neighborhood because that is where the available evidence is strongest and where new development work is currently happening.

Initial working model

These are not final claims for every Astrocade arcade game. They are the starting model for our Wizard of Wor class target and will be refined in the deeper chapters.

Memory shape

0000-3FFF  Program ROM, also Magic write window
4000-7FFF  Video RAM
8000-CFFF  Additional program ROM on larger games
D000-D03F  Protected RAM region
D040-DFFF  Work RAM / scratch RAM

Core I/O shape

00-0F  Video registers
10-1F  Sound / input reads
10-18  Sound writes
19     Expand register
78-7E  Pattern board writes
A55B   Protected RAM enable latch
Important: the same low address range can mean different things depending on whether the CPU is reading instructions or writing data. That is the heart of the Astrocade Magic system and one reason normal Z80 assumptions can mislead you.

How this guide will mark truth

Every low-level claim should carry a confidence tag. This keeps the guide honest and makes it easier to spot where we need another test, another trace, or a scope on real hardware.

TagMeaning
MAME-confirmedObserved in current MAME source or behavior.
Schematic-confirmedBacked by service manual schematic or board drawing.
ROM-observedSeen in original game disassembly or debugger trace.
Real-hardware-confirmedTested on an actual Astrocade arcade board/cabinet.
Working hypothesisLikely, useful, but not yet proven enough to teach as fact.
