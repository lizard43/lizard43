⟵ Intro ⌂ Build Environment ⟶

This page is the front door for actually doing things with the Bally/Midway Astrocade arcade platform: assembling code, running it in MAME, patching commercial ROM sets, replacing routines, and eventually proving the work on real card-rack hardware.

The hardware pages explain what the machine is. This development section explains how to work against it without treating it like a normal console, a normal Z80 computer, or a clean linear framebuffer.

Working target

The first target is practical replacement and extension work for Wizard of Wor, Gorf, Space Zap, and Robby Roto. Some pages will be general. Some will be game-specific.

Use this page as a table of contents for development notes. The linked pages can grow independently without turning the main Field Guide into one giant scroll.

Development tracks

There are two different workflows here. They overlap, but they are not the same job.

New code

Start with a clean reset vector, initialize the hardware, set up RAM and interrupts, draw through direct VRAM writes or Magic writes, then build upward into a complete arcade-targeted program.

Clean startup Known symbols Own ROM layout

ROM hacking / mods

Start from an existing commercial ROM set, disassembly, or MAME trace. Patch behavior, redirect routines, replace graphics/data, or splice new code into unused or repacked ROM space.

Existing code path Patch risk ROM rebuild
Rule of thumb New code is easier to reason about. ROM hacking is easier to ship as a small visible change. The best work usually borrows from both: build clean test programs first, then carry the proven routines into a patched commercial ROM.

Recommended workflow

The safest path is to move from small controlled tests toward larger changes. Do not start by rewriting half a commercial game and then wondering which subsystem broke.

Step Goal Proof
1 Build a tiny ROM that boots, clears video, and draws one known thing. MAME shows expected pixels; debugger reaches the main loop.
2 Add stack, RAM symbols, protected RAM writes, and a frame counter. Watchpoints show writes only where expected.
3 Add inputs and a visible debug status line. Coin, start, joystick, and trigger bits change on-screen.
4 Add graphics routines one at a time: direct VRAM first, Magic path second. Each primitive has a standalone test case.
5 Patch the same known-good routine into an existing ROM set. Original game still boots; only the intended behavior changes.
6 Move from MAME to real hardware only after emulator behavior is stable. EPROMs verify, board boots, and results match the MAME expectation.

Toolchain map

This section should stay opinionated. Use a small number of known-good tools and document the exact command lines that produce a bootable ROM set.

Assembler path

src/*.asm
  ↓
zmac
  ↓
.lst + .bin
  ↓
split / zip
  ↓
mame wow -rompath zout

Best for exact Z80 control, ROM patches, hardware register tests, and matching commercial disassembly style.

C path

src/*.c
  ↓
sdcc -mz80
  ↓
asm / ihx / bin
  ↓
arcade startup glue
  ↓
mame target test

Useful for higher-level experiments, but arcade startup and hardware assumptions must be explicit. Console Astrocade C examples are not drop-in arcade programs.

Types of mods

Not every mod has the same risk level. Keep the easy wins separate from deep code surgery.

Data-only edits Text, tables, colors, maze data, simple constants.
Graphics swaps Sprites, title art, font glyphs, tile/pattern replacements.
Routine patches Redirect a call/jump to a replacement routine with preserved registers.
Subsystem rewrites Input, drawing, sound, scoring, attract mode, or interrupt handling.
New game shell Clean reset path, own memory layout, own loop, own art/data pipeline.

Ground rules

Do this

Keep every experiment small enough to prove in isolation.
Write down the exact MAME command used for each test.
Save generated .lst, symbols, and ROM zip output.
Tag assumptions as MAME, disassembly, schematic, or real hardware.
Preserve registers when patching into existing commercial code.

Do not do this

Do not assume console Astrocade examples run on arcade hardware.
Do not write to low memory without knowing if you are hitting Magic.
Do not patch a ROM without keeping the original and a diffable trail.
Do not treat MAME timing behavior as final hardware proof.
Do not mix Wizard of Wor, Gorf, Space Zap, and Robby Roto maps blindly.
