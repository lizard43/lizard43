Video System

The arcade Astrocade video model from a developer’s point of view: color latches, palette split, commercial hi-res mode, the $4000 framebuffer, Magic writes, interlace field mirroring, and why the arcade screen is not just the console screen with a bigger hat.

The center of gravity is the arcade hi-res framebuffer

The most dangerous mistake is to approach Wizard of Wor / Gorf-class hardware as if it were mainly the home Bally Astrocade console video system. The shared custom-chip DNA matters, but the arcade target is dominated by a large, directly addressable, high-resolution framebuffer.

Main field-guide rule For arcade development, start with VRAM_SCREEN_START = $4000, SCREEN_STRIDE = $0050, SCREEN_WIDTH_PIXELS = 320, and SCREEN_HEIGHT_LINES = 204. Console low-res habits are supporting background knowledge, not the default programming model.

The current arcade include makes that bias explicit. It names VRAM_SCREEN_START / SCREEN_RAM as $4000, the screen end as $7FBF, and the high-resolution stride as $0050, which is 80 bytes per line. At 2 bits per pixel, 80 bytes gives 320 pixels per scanline.

The arcade include keeps the classic HVGLIB-style names, but also gives direction-aware names that are safer for arcade code. For video, the important output range is $00-$0F.

Direction still matters Some low ports have read-side aliases too, such as PORT_INT_STATUS at $08, PORT_VERT_FEEDBACK at $0E, and PORT_HORIZ_FEEDBACK at $0F. Do not assume an IN from a port has the same meaning as an OUT to that port.

Color latches and split palettes

The Astrocade video path is 2 bits per pixel, so each pixel selects one of four color indexes: 0, 1, 2, or 3. The arcade hardware exposes eight color latches: four for the right side and four for the left side. That is why the include names ports COL0R-COL3R and COL0L-COL3L.

; Example: program matching left/right palettes from a table
; A real project can choose different left/right values for split-screen effects.

ColorMap_Default:
        db      $51,$7C,$F3,$C7     ; right colors 0-3
        db      $51,$7C,$F3,$C7     ; left colors 0-3

LoadPalette8:
        ld      hl,ColorMap_Default
        ld      c,PORT_COL0R
        ld      b,8
.next:  outi                            ; write ports $00 through $07
        jr      nz,.next
        ret

For most new code, use a symmetric left/right palette first. Once the display is stable, then experiment with different left/right palettes and PORT_PAL_SPLIT. Keeping the two halves identical removes one moving part while debugging drawing code.

Color byte caution Console documentation is useful for understanding the color-byte format and hue/brightness idea, but the arcade field guide should document exact values from real WoW/Gorf/Robby code and real-screen observations. A value that looks correct in an emulator can still deserve a MAME-confirmed tag until checked on hardware.

Mode, height, and split controls

The next group after the color latches sets the video mode and timing-related behavior. In the include, PORT_VIDEO_MODE has the legacy alias CONCM, with constants VIDEO_MODE_CONSUMER = 0 and VIDEO_MODE_COMMERCIAL = 1. For arcade work, the commercial path is the important one.

Control Developer meaning Practical guidance
PORT_VIDEO_MODE / CONCM Selects commercial-style video behavior. Initialize early before drawing assumptions depend on hi-res layout.
PORT_PAL_SPLIT / HORCB Controls where the left/right palette boundary is applied. Use a safe/default value first; test split effects later.
PORT_HEIGHT / VERBL Vertical blanking / active screen height control. Use known WoW-style values until the chapter has schematic-confirmed details.
PORT_COLORMAP / COLBX Color-block multi-port behavior. Useful for compatibility with Astrocade-style color code, but explicit $00-$07 color latch writes are easier to reason about. 
; Conceptual video init shape.
; The exact values belong in project constants, tagged by source.

InitVideoCore:
        ld      a,VIDEO_MODE_COMMERCIAL
        out     (PORT_VIDEO_MODE),a

        ld      a,DEFAULT_PAL_SPLIT
        out     (PORT_PAL_SPLIT),a

        ld      a,DEFAULT_HEIGHT
        out     (PORT_HEIGHT),a
        ret

Direct framebuffer layout

The working arcade model is simple and powerful: video RAM begins at $4000. Each byte holds four 2-bpp pixels. Each high-resolution line is 80 bytes. The observed active screen area is 204 lines, for 16,320 bytes. The include captures this as:

VRAM_SCREEN_START       equ $4000
VRAM_SCREEN_END         equ $7FBF
VRAM_SCREEN_SIZE        equ $3FC0
SCREEN_RAM              equ VRAM_SCREEN_START
SCREEN_RAM_END          equ VRAM_SCREEN_END
SCREEN_BYTES            equ VRAM_SCREEN_SIZE
SCREEN_STRIDE           equ $0050     ; 80 bytes per scanline
SCREEN_WIDTH_PIXELS     equ 320
SCREEN_HEIGHT_LINES     equ 204

The reserved tail from $7FC0-$7FFF is intentionally named conservatively in the include. It is not treated as palette RAM in this arcade guide; color values are programmed through I/O ports instead.

$4000-$7FBF
active hi-res framebuffer, 320×204 working model
$7FC0-$7FFF
reserved tail, verify before using

Addressing pixels: X/Y to byte address

In hi-res mode, use byte X, not pixel X, when calculating the framebuffer address. Since each byte contains four pixels, xByte = xPixel / 4. The base address is:

address = SCREEN_RAM + (y * SCREEN_STRIDE) + (xPixel / 4)

For example, pixel coordinate x=200, y=20 lands at byte X 50, so the byte address is $4000 + 20*$50 + 50.

; zmac-style conceptual helper for an already byte-aligned X
; Input: D = y, E = xByte is possible, but this example keeps it readable.

; Constant expression example:
; upper-ish right debug mark at xByte 50, y 20
        ld      hl,SCREEN_RAM + (20 * SCREEN_STRIDE) + 50
Nibble/bit-plane caution The address gets you to the byte containing four pixels. Changing one pixel inside that byte means masking and replacing only its two-bit field. For quick solid boxes, byte-aligned drawing is much simpler and is the right first test.

Magic memory writes

Magic is the video accelerator path. In the machine model chapter, the key rule was that reads from low ROM fetch program bytes, while writes into the same low range feed the Magic hardware. The video chapter cares because Magic is how many fast draw operations become practical.

Register bit Include symbol Mask Meaning
0-1 MRSHFT $03 Shift amount field.
2 MRROT / MRROT_MASK $04 Rotate behavior.
3 MRXPND / MRXPND_MASK $08 Expand behavior, often paired with PORT_XPAND.
4 MROR / MROR_MASK $10 OR draw mode.
5 MRXOR / MRXOR_MASK $20 XOR draw/erase mode.
6 MRFLOP / MRFLOP_MASK $40 Flop/mirror behavior. 
; Example: set Magic to XOR mode before a Magic write path
        ld      a,MAGIC_XOR
        out     (PORT_MAGIC),a

; Optional expander state, if using expanded source data
        ld      a,XPAND_VALUE
        out     (PORT_XPAND),a

Magic deserves its own chapter because the behavior depends on current Magic register state, expander state, alignment, source bytes, and destination interaction. This video chapter only establishes where Magic lives in the video model.

Direct framebuffer writes vs Magic writes

Both paths are useful, but they solve different problems.

Path Use for Why Risk
Direct VRAM Clears, byte-aligned boxes, debug text, known pixel buffers. Easy to reason about: address plus byte value. Can be slower for shaped sprites and requires careful per-pixel masking.
Magic writes Sprites, masked/OR/XOR operations, shifted/flopped/expanded graphics. Hardware assists with common graphics transforms. Bad Magic state can corrupt unexpected screen regions.
Pattern board WoW/Gorf hardware pattern operations. Extra arcade-specific graphics hardware. Needs separate register discipline and schematic/ROM confirmation.
Debugging strategy Bring the screen up with direct VRAM first. Once palette, mode, stride, and field behavior are proven, then add Magic. Once Magic is stable, then bring in pattern-board acceleration.

Interlace and field mirroring

Our real-hardware notes point to a practical arcade issue: in high-resolution interlaced display, writing only one apparent framebuffer location may not update both visible fields the way you expect. The working rule has been to mirror certain screen writes at an offset of +$0800 when the real board needs both fields represented.

; Working-hypothesis pattern from real hardware experiments.
; Use only where the specific draw path requires field mirroring.

        ld      hl,SCREEN_RAM + (y * SCREEN_STRIDE) + xByte
        ld      (hl),a

        ld      de,$0800
        add     hl,de
        ld      (hl),a
Tag this carefully Field mirroring should be tagged real-hardware-observed or working hypothesis until each draw path is tested. MAME may not make the same artifact obvious, and schematics/service docs are needed to explain why the offset behaves the way it does.

This is another reason the field guide should not simply teach console Astrocade drawing. The arcade monitor, video RAM organization, and commercial mode timing are part of the actual software target.

Pattern board context

The pattern board is video-adjacent but should not be confused with the base Astrocade Magic register. The arcade include defines a separate block at $78-$7F:

PORT_PBLINADRL          equ $78
PORT_PBLINADRH          equ $79
PORT_PBCTL              equ $7A
PORT_PBROT              equ $7B
PORT_PBMASK             equ $7C
PORT_PBMR               equ $7D
PORT_PBXR               equ $7E
PORT_PBOR               equ $7F

The earlier I/O chapter treated $78-$7E as the MAME-backed pattern-board range. The include also names $7F as PORT_PBOR, which is useful for our project code but should stay marked for board/manual/ROM verification when describing exact behavior.

Port Likely role in project naming Field-guide caution
$78/$79 Pattern board line/address low/high. Confirm address loading sequence against WoW/Gorf routines.
$7A Control/status style register. Exact bits belong in the future pattern-board chapter.
$7B Rotation / alternate address alias in older code. Older aliases can obscure what the hardware is doing.
$7C Mask / high-address alias in older code. Prefer the clearer PBMASK naming until proven otherwise.
$7D Pattern-board Magic register. Do not confuse with Astrocade PORT_MAGIC at $0C.
$7E X register / width-style alias in older code. Needs examples from original ROM code.
$7F OR register in current include. Useful project symbol, but document exact MAME/manual support separately.

Practical video init sequence

A minimal arcade video bring-up should be boring. The goal is not to use every feature; the goal is to make a stable screen whose memory layout you trust.

; Suggested first-screen order for WoW-style experiments

InitVideoAndScreen:
        di

        ; 1. Select commercial/arcade video behavior.
        ld      a,VIDEO_MODE_COMMERCIAL
        out     (PORT_VIDEO_MODE),a

        ; 2. Program safe height and palette split values.
        ld      a,DEFAULT_HEIGHT
        out     (PORT_HEIGHT),a
        ld      a,DEFAULT_PAL_SPLIT
        out     (PORT_PAL_SPLIT),a

        ; 3. Program all 8 color latches.
        call    LoadPalette8

        ; 4. Set Magic to a harmless/direct baseline.
        xor     a
        out     (PORT_MAGIC),a
        out     (PORT_XPAND),a

        ; 5. Clear framebuffer directly first.
        call    ClearScreenDirect

        ; 6. Draw a byte-aligned test pattern or four corner boxes.
        call    DrawVideoSanityMarks

        ret

Only after this works should you add interrupts, Magic sprites, pattern-board operations, or split-palette tricks.

Rules of thumb

Do this

Use SCREEN_RAM + y*$50 + xByte for first-pass hi-res drawing.
Program all eight color latches before judging colors.
Use symmetric left/right palettes while debugging.
Start with direct VRAM writes before Magic writes.
Keep Magic, pattern-board, and direct-VRAM code paths clearly named.
Tag interlace/field behavior separately for MAME vs real hardware.

Do not do this

Do not assume console low-res layout is the arcade default.
Do not treat $7FC0-$7FFF as free RAM or palette RAM without proof.
Do not leave Magic in XOR/OR/expand mode accidentally.
Do not debug palette and drawing bugs at the same time if you can avoid it.
Do not confuse PORT_MAGIC $0C with pattern-board PBMR $7D.
Do not trust emulator-only video timing until the board agrees.
