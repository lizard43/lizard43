scope

This page describes the working development environment used by this guide: edit Z80 source, assemble it into arcade ROM output, then run or debug it with MAME.

Development can be done on Windows, macOS, or Linux. For consistency, the commands and examples in this guide assume a Linux shell. Windows users can adapt the same flow using WSL, MSYS2, Git Bash, PowerShell, or a native Windows MAME/zmac setup.

install zmac

zmac is the Z80 macro assembler used by many Astrocade examples. The important thing is to use the same assembler consistently, because syntax, macro behavior, and generated output can vary between assemblers. 

cd ~/src
git clone https://github.com/gp48k/zmac.git
cd zmac
make
sudo install -m 755 zmac /usr/local/bin/zmac
zmac -h

The zmac project also has built-in help/documentation through the command line, so keep that close while writing or porting assembly.

install and setup MAME

MAME is used both as the arcade hardware emulator and as the debugger. For this guide, the examples assume the executable is named mame and is available on the shell path.

distro package

Convenient and usually enough for quick testing, but it may lag behind the latest upstream MAME. 

sudo apt install mame
mame -version

first-time config

mame -cc
mame -showconfig | grep rompath

After configuration, point rompath at the directory containing clean original ROM ZIPs and any generated development ZIPs. For arcade work, the build script often passes -rompath explicitly so generated output can live beside the project instead of inside the global MAME ROM folder.

ROM naming matters Arcade drivers expect the ROM ZIP and files inside it to match the machine driver. A generated ZIP may build correctly and still fail if the filenames, sizes, or checksums are not what the driver expects.

MAME smoke tests

Before debugging custom code, verify that MAME can run the baseline machine and that the debugger opens. 

# Run a known-good arcade set.
mame wow -rompath ~/roms/mame -window -skip_gameinfo

# Run with debugger enabled.
mame wow -rompath ~/roms/mame -debug -window -skip_gameinfo

# Example with generated ROM output first, original ROMs second.
mame wow -rompath ./zout:~/roms/mame -debug -window -skip_gameinfo

tailor build.sh

The build script is the repeatable path from source code to a MAME run. Keep local paths near the top of the script and make the defaults easy to override from the shell. 

#!/usr/bin/env bash
set -euo pipefail

MAME_BIN="${MAME_BIN:-mame}"
ZMAC_BIN="${ZMAC_BIN:-zmac}"
ROMPATH="${ROMPATH:-$PWD/zout:$HOME/roms/mame}"
PROJECT="${PROJECT:-wow}"
MAIN_ASM="${MAIN_ASM:-wow.asm}"
OUT_DIR="${OUT_DIR:-zout}"

mkdir -p "$OUT_DIR"

# TODO: replace with the real assembler flags used by this project.
"$ZMAC_BIN" -o "$OUT_DIR/$PROJECT.bin" "$MAIN_ASM"

# TODO: package generated ROM files into the driver-compatible ZIP.

"$MAME_BIN" "$PROJECT" -rompath "$ROMPATH" -window -skip_gameinfo

local overrides

Let local machines override paths without editing the script every time. 

MAME_BIN=/opt/mame/mame ./build.sh
ROMPATH=./zout:/mnt/roms/mame ./build.sh

debug mode

Add a flag for MAME debugger launches so normal test runs and debugging use the same build path. 

./build.sh -debug
./build.sh -build

debugger basics

The MAME debugger is the fastest way to confirm where the CPU starts, what I/O ports are touched, and what RAM or video addresses are changing. 

# Break at reset/start code.
bpset 0000

# Disassemble around an address.
dasm 0100

# Show registers.
r

# Watch writes to an I/O port range.
wp 0:io,10000,w,(wpaddr & ff) < 20

Windows notes

Windows development is fine, but the examples in this guide stay Linux-oriented to avoid showing every command twice. The same pieces still apply: a Z80 assembler, MAME, the original ROM ZIPs, a repeatable build script, and a predictable output directory.

  • WSL: good for Linux-like scripts and command-line tools; be mindful of paths crossing between Windows and Linux filesystems.
  • Native Windows: works if MAME and zmac are installed as Windows executables and the build script is rewritten as .bat, PowerShell, or a portable Node/bash flow.
  • Path separators: the common failure point is usually not assembly code, but paths, ROM output locations, and where MAME expects ZIP files.
